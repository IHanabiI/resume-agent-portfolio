请基于候选人原始简历、STAR 证据、岗位 JD、匹配缺口、岗位对齐改写计划和用户补充回答，生成 TailoredResumeResult JSON。

核心工作方式：
- 你不是自由重写简历，而是按“岗位对齐改写计划”在原简历骨架内做定制。
- 优先执行 alignment_plan.required_actions、skill_adjustments、placeholders。
- 如果 alignment_plan 与原始事实冲突，以原始简历、STAR 证据和用户回答为准。
- 如果计划中某条没有证据支撑，不得写入正式简历，只能放入 still_missing_info 或 changelog。
- 使用“原简历结构骨架”理解章节、行号和列表结构。line_id / section_id 只用于定位，严禁输出到最终简历。
- “程序预排草稿”是系统根据 alignment_plan 做的保守顺序调整。最终 resume_markdown 应优先沿用它的章节与条目顺序，再在事实范围内改写措辞。
- 不得在最终简历中输出“程序预排草稿”“岗位对齐计划”“alignment_plan”等内部工作痕迹。

必须遵守：
- 最终简历不能编造事实。
- 不得把 JD 中的要求直接写成候选人能力。
- 用户只说“了解”，不能写成“熟练”或“精通”。
- 用户只说“参与”，不能写成“负责”或“主导”。
- 没有量化数据时，不能添加百分比、金额、人数、增长率、效率提升幅度。
- 可以优化表达，但不能改变事实。
- 用户回答“没有 / 不清楚 / 跳过”的内容不能进入正式简历正文。
- alignment_plan.do_not_use_claims 中的内容不能写进 resume_markdown。

参考项目式三项下限：
- 项目/经历排序：同一顶层章节内，把与 JD 最相关且有证据支撑的项目/经历放到更醒目的位置；没有把握时保持原顺序并在 changelog 说明。
- 项目内成果排序：同一个项目/经历内部，把最贴合 JD 的行动或结果要点放在前 1-2 条。
- 技能排序和措辞：专业技能板块优先展示 JD 需要且候选人已有证据的技能；缺失技能不能写成已掌握。

结构冻结：
- 只改内容不改架子。保留原简历主要章节，不新增“交付说明/证据/分析/补充信息/改动说明”类章节。
- 原简历已有的工作经历、项目经历、教育经历不要删除；低相关内容可以后移、压缩或弱化。
- 列表格式保持一条一行，不把多条列表项压成一段普通文字。
- 原本是项目/经历条目的内容，应继续作为项目/经历条目，不要改成大段说明。
- 尽量保留原简历的标题层级和章节顺序；只有 alignment_plan 明确要求重排时，才在同一章节内调整条目顺序。
- 如果原简历某章节只有普通段落，不要强行改成大量标题；如果原简历是列表，输出也应保持列表。

输出要求：
- resume_markdown：完整定制简历正文，Markdown 格式，只能包含投递时会出现在简历里的内容。
- opener_markdown：给 HR/招聘方的第一条沟通开场白，Markdown 格式；只能引用简历、记忆库、GitHub 或用户回答中存在的事实。
- changelog_markdown：改动说明，Markdown 格式；逐条说明改了哪里、为什么改、依据哪个 JD 要求；如果有缺失数据，使用 `[请填写：xxx]`；如果是轻度推断，使用 `[需用户确认：xxx]`。
- optimization_notes：说明做了哪些优化。
- integrated_keywords：列出自然融入的 JD 关键词。
- still_missing_info：仍建议补充的信息。
- evidence_map：每条关键简历内容对应来源。

resume_markdown 额外约束：
- 不要输出“补充信息”“记忆库可用事实”“GitHub 相关证据”“证据来源”“改动说明”“优化说明”“待确认”等交付说明章节。
- 用户补充回答、记忆库、GitHub 只能作为事实来源，用于改写工作经历、项目经历、技能或个人优势，不能整段原样堆到简历里。
- 标题层级必须克制：`#` 只用于姓名或简历主标题；`##` 只用于顶层章节；`###` 只用于具体公司、项目、学校或经历条目。详细说明、职责、成果必须用普通段落或列表，不得作为标题。
- 列表项应短而可扫读。过长说明请拆成 1-3 条要点，避免整段背景说明成为一个巨大列表项。
- 保留简历常见结构，例如个人信息、个人优势、专业技能、项目经历、工作经历、教育背景；不要把 changelog 或证据表混入正文。
- 对空壳经历不能补写虚构内容；如果岗位相关但缺内容，保留原条目并插入 `[请填写：需要补充的具体经历/结果]` 或在 still_missing_info 中提醒。
- 对轻度推断内容必须保留 `[需用户确认：xxx]`，不能写成确定事实。
- 最终 resume_markdown 不能出现 `L001`、`S001`、`line_id`、`section_id` 等内部定位标记。
- 最终 resume_markdown 不能出现“程序预排草稿”“原简历结构骨架”“岗位对齐计划”等内部过程标题。
